## 1. 产品简介

**PromptScope** 是一套面向对话式 AI 产品的自动化指令遵循评测工具。

传统的人工质检方式成本高、周期长、评分标准难以统一。PromptScope 通过模拟真实用户与您的对话模型进行多轮交互，再由专业评判模型从 7 个维度打分，最终生成一份**可量化、可追溯、可对比**的评测报告。

**一句话：把原来需要人工听几百通对话才能给出的质检结论，压缩到几分钟内自动完成。**

## 2. 适用场景

| 使用场景 | 您想解决的问题 | 本系统的作用 |
|---------|-------------|------------|
| 上线前验收 | 新版本模型是否符合任务指令要求？ | 跑完整评测，给出量化通过/失败结论 |
| 版本对比 | A 版本和 B 版本哪个更好？ | 分别评测，直接比较报告得分 |
| 问题诊断 | 模型哪里做得不够好？ | 报告中的检查点详情精确定位薄弱环节 |
| 持续监控 | 模型更新后是否出现退步？ | 定期评测，跟踪历史得分趋势 |
| 指令调优 | 修改任务指令后效果变化？ | 快速验证修改效果，缩短调优周期 |
| 生产对话回检 | 对真实用户的历史对话进行评测？ | 通过 `/eval/evaluate` 接口直接提交对话记录 |

## 3. 核心能力一览

### 3.1 全自动对话模拟

系统采用 **4 维度组合 Persona 系统**，从以下 4 个独立轴向组合出多样化的用户行为：

| 维度 | 可选值 | 含义 |
|------|-------|------|
| **配合意愿** | 积极配合 / 态度中立 / 抵触抗拒 | 用户对对话目标的配合程度 |
| **话量风格** | 健谈多话 / 寡言少语 / 刨根问底 / 敷衍应付 | 用户的表达方式 |
| **业务熟悉度** | 完全新手 / 资深老手 / 一知半解 | 用户对业务背景的了解程度 |
| **时间紧迫度** | 从容不迫 / 急迫匆忙 | 用户在对话中的耐心 |

4 个维度的组合理论上可生成最多 **72 种不同 Persona**，默认运行 **14 种**变体（可通过参数调整）。传统 7 种原型（配合型、抗拒型、困惑型、急躁型、跑题型、沉默型、细节追问型）在此系统中作为维度组合的标准映射保留，可继续使用。

### 3.2 支持两种被测 Agent 模式

| 模式 | 参数值 | 说明 |
|------|--------|------|
| **LLM 模拟** | `agent_type: "llm"` | 用任意 LLM API 模拟对话模型，适合开发调试阶段 |
| **Friday 数字人** | `agent_type: "friday"` | 直接接入 Friday 数字人 API，评测真实生产版本 |

### 3.3 7 维度独立评分

系统从 **7 个维度**独立评判，并按权重加权汇总为综合得分（0–100）：

| 维度 | Key | 默认权重 | 评测内容 |
|------|-----|---------|---------|
| 流程遵循 | `flow` | 25% | 是否按任务指令规定的流程节点推进 |
| 约束合规 | `constraint` | 20% | 是否严格遵守硬性约束规则 |
| 知识准确 | `knowledge` | 20% | 业务信息是否与指令中的知识要点一致 |
| 风格语气 | `style` | 10% | 表达风格是否符合角色定义 |
| 连贯性 | `coherence` | 10% | 上下文逻辑一致，无答非所问或前后矛盾 |
| 安全合规 | `safety` | 10% | 无有害内容、无系统提示泄露、无幻觉编造 |
| 应变能力 | `adaptability` | 5% | 面对异常用户行为时的处理质量 |

**综合得分 = Σ（维度分 × 权重）× 100**

> **硬约束惩罚机制：** 若任意硬约束违规（得分 < 50%），综合得分上限自动降至 **60 分**，无论其他维度表现多优秀。

### 3.4 自洽性检验

主观维度（连贯性、安全、应变等）默认进行 **3 次独立评判**（`SELF_CONSISTENCY_RUNS`），取一致性结论作为最终得分，有效降低随机性。每次评判还附带置信度（`confidence`），低置信度时自动触发异常标记。

### 3.5 异常检测

系统自动检测以下异常情况并在报告中标注：
- 评分异常偏高（`SUSPICIOUS_FULL_SCORE`）
- 评分呈双峰分布（`BIMODAL`）
- 置信度偏低（`LOW_CONFIDENCE`）
- 证据验证失败（`EVIDENCE_WARNINGS`）

### 3.6 证据可追溯

报告中每一条评分结论都附带原始对话的轮次引用和原文摘录（`Evidence`）。您可以直接看到"模型在第几轮说了什么"导致了某项扣分。

### 3.7 多格式报告输出

每次评测自动生成三种格式：
- **HTML 报告**：含可视化图表，适合人工阅读和分享
- **Markdown 报告**：适合在文档系统中粘贴存档
- **JSON 数据**：适合接入自研系统或进行批量数据分析

### 3.8 内置 AI 助手

Web 界面内置 **PromptScope AI 助手**，可在报告页直接提问，助手能结合当前报告数据（得分、失败项、改进建议）给出具体可操作的分析。助手使用独立的 LLM Key（`ASSISTANT_LLM_*`），与评判/模拟密钥隔离。

### 3.9 指令优化建议

评测完成后，系统可根据薄弱维度和关键失败项，自动生成针对原始任务指令的优化建议（通过 `/eval/optimize-prompt` 接口或 Web 界面的「优化建议」功能）。

## 4. 快速上手

### 4.1 准备工作：安装依赖

```bash
pip install -r requirements.txt
```

### 4.2 准备工作：配置环境

复制 `.env.example` 为 `.env`，填入以下信息：

```env
# 评判模型（推荐 GPT-4o 或同等能力模型）
EVAL_LLM_API_KEY=your_eval_api_key
EVAL_LLM_BASE_URL=https://api.openai.com/v1
EVAL_LLM_MODEL=gpt-4o

# 用户模拟模型（可用较低成本模型）
SIM_LLM_API_KEY=your_sim_api_key
SIM_LLM_BASE_URL=https://api.openai.com/v1
SIM_LLM_MODEL=gpt-4o-mini

# 被测的对话模型（LLM 模拟模式）
AGENT_LLM_API_KEY=your_agent_api_key
AGENT_LLM_BASE_URL=https://api.openai.com/v1
AGENT_LLM_MODEL=your_model_name

# 内置 AI 助手（留空则 fallback 到 EVAL_LLM_*）
ASSISTANT_LLM_API_KEY=your_assistant_api_key
ASSISTANT_LLM_BASE_URL=https://api.openai.com/v1
ASSISTANT_LLM_MODEL=gpt-4o-mini

# Friday 数字人 API（可选，仅 friday 模式需要）
FRIDAY_API_URL=https://your-friday-endpoint
FRIDAY_API_KEY=your_friday_key
FRIDAY_LLM_MODEL=friday-model-name
```

### 4.3 方式一：Web 控制台（推荐）

1. 构建前端（首次运行）：
   ```bash
   cd web && npm install && npm run build
   ```
2. 启动服务：
   ```bash
   python cli.py serve --port 8000
   ```
   开发模式（前端热更新）：
   ```bash
   python cli.py serve --port 8000 --dev
   ```
3. 浏览器打开 `http://localhost:8000`
4. 在「新建评测」页面粘贴任务指令
5. 选择要测试的 Persona 组合和评测参数
6. 点击「开始评测」，实时查看进度流，完成后查看报告

### 4.4 方式二：命令行

```bash
# 完整评测（使用所有 Persona）
python cli.py run --instruction my_task.txt

# 仅测试部分 Persona 原型
python cli.py run --instruction my_task.txt --personas cooperative,resistant,impatient

# 仅解析指令结构（不运行评测）
python cli.py parse --instruction my_task.txt

# 模拟单次对话（不评分，用于调试）
python cli.py simulate --instruction my_task.txt --persona resistant --max-turns 20
```

### 4.5 方式三：REST API

`/eval/run` 为 **SSE 流式接口**，通过 `text/event-stream` 实时推送进度事件，而非返回 Job ID 后轮询。

```bash
# 提交评测任务（SSE 流式，实时推送进度）
curl -N -X POST http://localhost:8000/api/eval/run \
  -H "Content-Type: application/json" \
  -d '{"instruction": "你是一名外呼专员...", "agent_type": "llm"}'

# 直接评测已有对话（无需模拟，同步返回）
curl -X POST http://localhost:8000/api/eval/evaluate \
  -H "Content-Type: application/json" \
  -d '{
    "instruction": "你是...",
    "dialogue": [
      {"role": "agent", "content": "您好，我是小美..."},
      {"role": "user", "content": "嗯，什么事？"}
    ]
  }'

# 获取报告（json/html/md）
curl http://localhost:8000/api/eval/report/{report_id}?fmt=html

# 列出最近 20 份报告
curl http://localhost:8000/api/eval/reports

# 删除报告
curl -X DELETE http://localhost:8000/api/eval/report/{report_id}
```

## 5. 理解评测报告

### 5.1 报告结构速览

```
┌─────────────────────────────────┐
│  执行摘要                        │  ← 综合得分、对话数量、关键失败数
├─────────────────────────────────┤
│  7 维度得分明细                   │  ← 每个维度的得分和进度条
├─────────────────────────────────┤
│  各 Persona 得分分布              │  ← 多种用户类型的独立得分
├─────────────────────────────────┤
│  关键失败（硬约束违规）             │  ← 触发总分上限的严重问题
├─────────────────────────────────┤
│  改进建议                        │  ← 系统自动生成的优先级建议
├─────────────────────────────────┤
│  根因分析（Layer 6）              │  ← 归因到知识库/Prompt/工具/流程等类别
└─────────────────────────────────┘
```

### 5.2 如何读懂综合得分

| 分数区间 | 含义 | 建议行动 |
|---------|------|---------|
| 90–100 | 优秀，指令遵循接近完整 | 可上线，持续监控 |
| 75–89 | 良好，存在少量问题 | 针对薄弱维度调优后上线 |
| 60–74 | 及格，存在明显缺陷 | 重点修复后重新评测 |
| < 60 | 不及格或触发硬约束惩罚 | 需要较大幅度改进 |

> 若综合得分恰好 ≤ 60 且 `hard_constraint_failed: true`，说明是因硬约束违规触发了分数上限，需优先修复约束问题。

### 5.3 维度得分含义

- **流程遵循**：检测模型是否按照指令中的对话流程图执行，每个必须节点（`required: true`）权重为 2.0，可选节点权重为 1.0
- **约束合规**：分为硬约束（`hard`，权重 3.0）和软约束（`soft`，权重 1.5）。字数限制用合规率计算，禁用词用二值判断
- **知识准确**：每个知识点独立检测（权重 1.5），判断表述是否与 `correct_content` 一致
- **风格语气**：评判语气是否符合指令定义的角色风格（1–5 分制评分）
- **连贯性**：两个全局检查点，分别评估上下文逻辑一致性和语义相关性
- **安全合规**：三个全局检查点，检测系统信息泄露、有害内容和幻觉编造
- **应变能力**：两个全局检查点，评估处理异常用户行为和修复误解的能力

### 5.4 三层分析链

评测完成后，系统依次运行三层 AI 分析：

| 层次 | 名称 | 说明 |
|------|------|------|
| Layer 4 | 单对话分析（ReportAgent） | 对每条对话独立总结优势、弱点和改进建议 |
| Layer 5 | 横向对比（JudgeAgent） | 跨对话对比，排出 Top5/Bottom5，识别共性问题 |
| Layer 6 | 根因归因（AttributionAgent） | 将 Bottom5 的失败追溯到根本原因（知识库缺失、Prompt 问题、工具失效、流程缺陷、约束过严等） |

### 5.5 如何定位具体问题

**Step 1：** 找到维度得分最低的项（如"流程遵循"只有 55%）

**Step 2：** 查看该 Persona 的具体对话记录，找到对话偏离流程的轮次

**Step 3：** 结合证据原文（`Evidence.quoted_text`）判断是模型的问题还是指令描述不清

**Step 4：** 参考 Layer 6 根因分析中的 `fix_suggestion`，针对性修改指令或模型

**Step 5：** 重新提交评测，对比分数变化

## 6. 配置与调参

### 6.1 调整维度权重

在 `.env` 文件中修改权重（**七项之和无需严格等于 1.0，系统会自动归一化**）：

```env
# 示例：外呼场景，重流程和约束
WEIGHT_FLOW=0.30
WEIGHT_CONSTRAINT=0.25
WEIGHT_KNOWLEDGE=0.20
WEIGHT_STYLE=0.10
WEIGHT_COHERENCE=0.08
WEIGHT_SAFETY=0.05
WEIGHT_ADAPTABILITY=0.02

# 示例：客服场景，重安全和知识
WEIGHT_FLOW=0.20
WEIGHT_CONSTRAINT=0.20
WEIGHT_KNOWLEDGE=0.25
WEIGHT_STYLE=0.10
WEIGHT_COHERENCE=0.10
WEIGHT_SAFETY=0.10
WEIGHT_ADAPTABILITY=0.05
```

### 6.2 调整 Persona 数量和维度

通过 API 或 Web 界面，可以选择具体的维度值组合，系统取笛卡尔积生成测试矩阵：

```json
{
  "instruction": "...",
  "dimensions": {
    "cooperation": ["cooperative", "resistant"],
    "verbosity": ["verbose", "terse"],
    "familiarity": ["novice"],
    "urgency": ["relaxed"]
  }
}
```

默认（不传 `dimensions`）：使用所有维度组合，共生成 `PERSONA_VARIANT_COUNT`（默认 14）种变体。

也可以通过 `/eval/generate-personas` 接口，基于任务指令内容让 AI 自动生成有针对性的 Persona，再传入 `/eval/run`。

### 6.3 调整对话最大轮次

```env
MAX_TURNS=30   # 默认 30 轮，可根据实际通话时长调整（范围建议 10–50）
```

### 6.4 自洽性检验强度

```env
SELF_CONSISTENCY_RUNS=3    # 主观维度重复评判次数，越高越稳定但 API 调用成本越高
CONFIDENCE_THRESHOLD=0.6   # 置信度低于此值触发低置信度异常标记
```

### 6.5 并发与限流控制

```env
SIM_CONCURRENCY=8          # 并发对话模拟数量
EVAL_CONCURRENCY=8         # 并发评判数量
EVAL_BATCH_SIZE=12         # 每批处理的 Persona 数量（影响断点续跑粒度）

# LLM 限流（适用于有严格 RPS 限制的服务商，如 DeepSeek）
LLM_RATE_LIMIT_RPS=0.0     # 0 = 禁用；设置为 5.0 即限制每秒 5 个请求
LLM_RATE_LIMIT_BURST=10    # 令牌桶突发容量
LLM_MAX_RETRIES=5          # 请求失败后的最大重试次数
```

### 6.6 内置 AI 助手配置

```env
# 推荐配置低成本模型，与评判模型隔离
ASSISTANT_LLM_API_KEY=your_key
ASSISTANT_LLM_BASE_URL=https://api.openai.com/v1
ASSISTANT_LLM_MODEL=gpt-4o-mini
```

留空时自动 fallback 到 `EVAL_LLM_*`。助手接口有独立的速率限制（每 IP 每分钟 20 次），且对输入进行注入检测和长度截断（2000 字符上限）。

## 7. API 参考

### 基础信息

- 服务地址：`http://localhost:{port}/api`
- 所有请求均为 JSON 格式
- `/eval/run` 为 SSE 流式接口（`text/event-stream`），其余接口为同步 JSON 响应

### 接口列表

| 方法 | 路径 | 说明 |
|------|------|------|
| `GET` | `/health` | 服务健康检查 |
| `POST` | `/eval/parse` | 解析任务指令，返回结构化 Spec |
| `POST` | `/eval/run` | 提交完整评测任务（SSE 流式，实时推送进度） |
| `POST` | `/eval/evaluate` | 对外部提供的对话进行评测（无需模拟，同步返回） |
| `POST` | `/eval/simulate` | 模拟单次对话（不评分，同步返回） |
| `POST` | `/eval/generate-personas` | 基于指令内容 AI 生成测试 Persona |
| `POST` | `/eval/optimize-prompt` | 根据评测结果生成指令优化建议 |
| `GET` | `/eval/reports` | 列出最近 20 份报告摘要 |
| `GET` | `/eval/report/{report_id}` | 获取报告（`?fmt=json/html/md`） |
| `DELETE` | `/eval/report/{report_id}` | 删除报告及关联文件 |
| `POST` | `/assistant/chat` | 内置 AI 助手（SSE 流式，可注入报告上下文） |

### POST /eval/run 请求体

```json
{
  "instruction": "任务指令全文",
  "personas": ["cooperative", "resistant"],
  "dimensions": {
    "cooperation": ["cooperative", "resistant"],
    "verbosity": ["verbose", "terse"],
    "familiarity": ["novice", "expert"],
    "urgency": ["relaxed"]
  },
  "generated_personas": null,
  "agent_type": "llm",
  "friday_api_url": "",
  "friday_model": "",
  "eval_llm_api_key": "",
  "eval_llm_base_url": "",
  "eval_llm_model": "",
  "sim_llm_api_key": "",
  "sim_llm_base_url": "",
  "sim_llm_model": "",
  "agent_llm_api_key": "",
  "agent_llm_base_url": "",
  "agent_llm_model": ""
}
```

> `personas`（原型名列表）和 `dimensions`（维度矩阵）二选一；`generated_personas` 用于传入 `/eval/generate-personas` 返回的自定义 Persona 列表。LLM 配置项留空时使用 `.env` 中的值，**三项（key/url/model）须同时填写才生效**。

### POST /eval/run SSE 事件格式

```jsonc
// 阶段变更
{"type": "phase", "phase": "parsing|enriching|simulating|evaluating|analyzing|report", "message": "..."}

// 单条对话完成（实时推送）
{"type": "dialogue", "dialogue_id": "...", "persona_label": "...", "score": 82.4, "turns": [...], "termination_reason": "natural_end"}

// 评测出错（单条）
{"type": "eval_error", "error": "..."}

// 全部完成
{"type": "done", "report_id": "a1b2c3d4", "overall_score": 82.4, "paths": {...}, "run_id": "..."}

// 全局错误
{"type": "error", "error": "错误信息"}
```

### POST /eval/evaluate 请求体

```json
{
  "instruction": "任务指令全文",
  "dialogue": [
    {"role": "agent", "content": "您好，我是..."},
    {"role": "user",  "content": "嗯，什么事？"}
  ],
  "eval_llm_api_key": "",
  "eval_llm_base_url": "",
  "eval_llm_model": ""
}
```

### POST /eval/generate-personas 请求体

```json
{
```

## 8. 常见问题

**Q：评测一次需要多长时间？**  
A：取决于 Persona 数量和最大轮次设置。默认配置（14 种 Persona，最大 30 轮，`SIM_CONCURRENCY=8`）通常需要 5–20 分钟。只测试 7 种原型约需 5–12 分钟。

**Q：评测结果的可信度如何？**  
A：主观维度使用 3 次自洽性检验，并附带置信度指标。客观类评分（字数、禁用词等）为确定性计算，结果 100% 可重现。建议将综合得分作为参考，结合改进建议的具体证据做判断。

**Q：同一份指令每次评测结果会完全一样吗？**  
A：客观维度（约束合规中的字数、禁用词部分）结果一致。主观维度存在一定波动（±3–5 分属正常范围）。多次评测取平均可获得更稳定的基线。

**Q：评测中途中断了怎么办？**  
A：系统每完成一批对话后会自动写入 `checkpoint.json`。重新运行相同的命令或重新提交相同指令时，会自动跳过已完成的对话，从上次断点继续。

**Q：Friday 模式和 LLM 模式有什么区别？**  
A：LLM 模式使用您配置的任意 LLM API 作为被测 Agent，适合开发调试。Friday 模式直接调用 Friday 数字人的实际 API，评测真实上线版本。如果 Friday 配置不完整，系统会自动 fallback 到 LLM 模式。

**Q：可以评测真实用户的历史对话记录吗？**  
A：可以。使用 `POST /eval/evaluate` 接口，直接提交对话记录（agent/user 轮次列表）和任务指令，系统会跳过模拟步骤，直接对该对话进行多维度评判。

**Q：指令解析失败怎么办？**  
A：先运行 `python cli.py parse --instruction my_task.txt` 查看解析结果。如果流程节点解析有误，建议在指令中使用更明确的结构标识（如编号列表、"步骤一/步骤二"等），或添加"对话流程："等显式标题。

**Q：API Key 如何管理？三个模型可以用同一个 Key 吗？**  
A：可以。如果三个 LLM 使用同一服务商，填写同一个 Key 即可。被测 Agent 的 Key 单独配置，便于后续切换模型进行对比评测。

**Q：维度权重总和不等于 1.0 会有问题吗？**  
A：不会。系统在计算综合得分时会自动对权重归一化，因此权重只需保持相对比例正确即可。

**Q：使用 DeepSeek 等有 RPS 限制的服务商时如何避免限流报错？**  
A：在 `.env` 中设置 `LLM_RATE_LIMIT_RPS=5.0`（根据您的配额调整）和 `LLM_RATE_LIMIT_BURST=10`，系统会自动限速。同时适当降低 `SIM_CONCURRENCY` 和 `EVAL_CONCURRENCY`。

**Q：AI 助手提示"LLM 未配置"怎么办？**  
A：在 `.env` 中配置 `ASSISTANT_LLM_API_KEY`，或确保 `EVAL_LLM_API_KEY` 已填写（助手会自动 fallback）。

**Q：`/eval/run` 接口为什么没有 Job ID？**  
A：该接口使用 SSE（Server-Sent Events）流式协议，连接建立后实时推送进度事件，直到收到 `{"type":"done",...}` 表示完成。无需轮询 Job 状态。

## 9. 名词解释

| 术语 | 含义 |
|------|------|
| **任务指令** | 给对话模型的完整提示词，定义其角色、对话流程和约束规则 |
| **Persona（用户角色）** | 系统模拟的虚拟用户，由 4 个维度的组合确定行为特征 |
| **Persona 维度** | 配合意愿 / 话量风格 / 业务熟悉度 / 时间紧迫度，4 个独立轴向 |
| **InstructionSpec** | 任务指令的结构化解析结果，包含流程图、约束列表和知识点列表 |
| **检查点（Checkpoint）** | 从任务指令中拆解出的单条可验证评估项，有权重和评测类型 |
| **硬约束（Hard Constraint）** | 必须严格遵守的规则（如禁用词），违反时触发总分上限 60 分惩罚 |
| **软约束（Soft Constraint）** | 程度性要求（如"语气亲切"），通过 LLM 评分，允许部分浮动 |
| **自洽性检验（Self-Consistency）** | 对同一检查点进行多次独立评判，取一致性结论提高可靠性 |
| **置信度（Confidence）** | 评判结论的可靠程度（0–1），低于阈值时自动标记为异常 |
| **证据（Evidence）** | 支撑某项评分结论的具体对话轮次编号和原文引用 |
| **异常标记（Anomaly Flag）** | 系统自动检测的评分异常情况，如全分异常、双峰分布等 |
| **Layer 4/5/6 分析链** | 评测完成后的三层 AI 分析：单对话分析 → 横向对比 → 根因归因 |
| **ReportAgent** | Layer 4，对每条对话独立分析优势和弱点 |
| **JudgeAgent** | Layer 5，跨对话横向对比，排出 Top5/Bottom5，识别共性问题 |
| **AttributionAgent** | Layer 6，将 Bottom5 失败追溯到根本原因并给出修复建议 |
| **Friday 模式** | 直接接入 Friday 数字人 API 的评测模式 |
| **SSE（Server-Sent Events）** | `/eval/run` 使用的实时推送协议，无需轮询即可获取进度 |
| **断点续跑（Checkpoint Resume）** | 评测中途中断后，重新运行时自动跳过已完成对话的机制 |
